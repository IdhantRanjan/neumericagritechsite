# Security — Neumeric platform

Farm boundaries, claim evidence, and marketing positions are financial data.
This document states how the platform protects them, what's been verified,
and what's still open. Honest by design: gaps are listed, not hidden.

## Authentication & sessions

- **Magic-link email auth** (primary): 256-bit single-use tokens, 15-minute
  expiry, only the SHA-256 of the token stored — a database leak cannot mint
  sessions. **Password fallback** (optional per user): scrypt (N=16384, r=8,
  p=1, 32-byte salt), constant-time comparison, never logged.
- Sessions: 256-bit tokens (hashed at rest), httpOnly + SameSite=Lax +
  Secure cookies, 30-day expiry, per-device revocation and
  "sign out everywhere". Every auth event is written to `login_events`
  (per-account audit log visible to the user at /account).
- Legacy pilot workspaces use unguessable bearer links (192-bit); owners are
  prompted to attach an email account; links are rotatable.

## Tenant isolation & RBAC

- Every read helper and mutation is scoped to the resolved operation;
  entity lookups verify ownership before returning rows.
- Roles: `owner` / `member` / `advisor` (read + outcomes) / `partner`
  (read-only, reserved for the lender channel) / `visitor` (public demo,
  read-only — all writes are blocked server-side).
- **Verified:** `platform/scripts/test-isolation.ts` — 14 checks covering
  cross-tenant reads, forged operation-cookie resolution, bearer-token
  guessing, the full role permission matrix, and demo read-only. Run it in CI
  and before every release.

## Input validation & abuse protection

- Server-side validation on all mutations (zod on auth/waitlist; explicit
  bounds checks on money-adjacent numbers; GeoJSON boundary geometry is
  parsed, range-checked, and re-measured before storage).
- DB-backed fixed-window rate limits on every public endpoint (magic links
  per-IP and per-email, password attempts, invites, waitlist, token
  verification) — works across serverless instances.
- Upload limits (25 MB), filename sanitization, content-addressed storage
  (the SHA-256 is the key, so path traversal has no surface).

## OWASP top-10 posture

| Risk | Posture |
| --- | --- |
| Injection | Drizzle parameterized queries only; no string-built SQL |
| Broken auth | Hashed tokens, rate limits, single-use links, revocation |
| Sensitive data exposure | TLS everywhere (Vercel/Turso/Blob); no secrets in repo; structured logs exclude password/token values |
| XSS | React escaping; CSP with no external script origins; no `dangerouslySetInnerHTML` |
| Broken access control / IDOR | Operation-scoped lookups + isolation test suite |
| CSRF | SameSite=Lax cookies; Next server actions use origin-checked POSTs |
| SSRF | Outbound fetches go only to pinned hosts (Earth Search STAC, the open Sentinel-2 S3 bucket, Open-Meteo, Resend, Vercel Blob); no user-supplied URLs are fetched |
| Security misconfig | CSP, HSTS, nosniff, frame-ancestors 'none', Referrer-Policy set in next.config.ts |
| Vulnerable deps | `npm audit` at last release: 6 moderate, all in dev-only/transitive tooling (esbuild via drizzle-kit's dev toolchain; postcss pinned inside Next). None reachable in the production bundle; re-check each release until upstream patches land |
| Logging/monitoring | Structured JSON logs + instrumentation.onRequestError; /api/health for uptime checks |

## Evidence integrity (why an insurer can trust the records)

- Every money-relevant artifact (imagery capture, field condition record,
  trigger evaluation, ground-truth label) appends to a tamper-evident
  hash chain (`provenance_entries`): each entry commits to the previous
  entry's hash and is HMAC-signed with a server-held key. Altering any
  historical record breaks every later hash.
- Satellite analyses store scene ids, asset reference hashes, methodology
  version hashes, and full calculation traces — reproducible byte-for-byte.
- Documented next step: external RFC-3161 / OpenTimestamps anchoring so
  capture time is verifiable without trusting Neumeric.

## Data durability & backups

- Primary store: Turso (hosted libSQL). Point-in-time restore is available
  on Turso's paid plans (**verify plan tier before real-farmer scale** — the
  free tier has limited retention; documented as a launch dependency in
  docs/DEPLOY.md).
- Imagery bytes: Vercel Blob, content-addressed by SHA-256 — re-uploadable
  and verifiable against the hashes stored in the DB.
- Migrations are forward-only, generated by drizzle-kit, and run
  automatically at boot; test against a copy of prod schema before deploy.

## Known gaps (tracked, not hidden)

1. **Email delivery requires RESEND_API_KEY** — until set, magic links can't
   send in production (legacy links still work). Launch dependency.
2. Turso backup tier needs confirmation before scale (above).
3. No 2FA yet; magic-link-only accounts inherit the security of the user's
   inbox. Acceptable at pilot scale; revisit before broad launch.
4. CSP retains 'unsafe-inline' for script (Next hydration payload); nonce-based
   CSP is the upgrade path.
5. No third-party penetration test yet — recommended before public launch.
